# 计算器内容增强统一方案（避免模板化/凑字）

目标：以“计算器页”为内容根基持续扩容，同时避免出现**同结构、同节奏、同字数**的模板化文本（容易被判低价值）。

## 1) 底线与原则

- **底线**：所有计算器内容块必须可独立解决用户问题；不写“看似很多但信息密度很低”的段落。
- **不要追求统一字数**：`minWords=500` 只是“薄内容防线”，不是目标字数。
- **每页必须有独有信息**：参数、单位、典型场景、易错点、购买/施工限制，必须和该计算器强绑定。
- **避免通用模板**：不要每篇都按相同标题/同样顺序/同样段落长度来写。

## 2) 字数策略（用区间，不用固定值）

把页面分成 3 档，混合使用：

- **A 档（短但密）**：`550–800` 词：适合“单一参数、用户决策少”的计算器（例如袋装换算类）。
- **B 档（主力）**：`800–1100` 词：大多数计算器目标区间。
- **C 档（长且深）**：`1100–1600` 词：多场景、多坑、多材料/系统差异的计算器（例如地面系统、屋面、混凝土、排水/基层相关）。

说明：我们不追求所有页面都进同一档；**分布越自然越好**。

## 3) “内容模块库”（每页选 3–5 个，且顺序可变）

每个计算器在原有必备数组（quick/howToMeasure/assumptions/buyingTips/commonMistakes）之外，优先从以下模块里组合：

- **Worked example（示例）**：给一组具体数字，展示“怎么填/怎么解释结果/怎么加 buffer”。
- **Decision table（决策表）**：按场景给推荐（厚度、损耗、材料类型、是否分层估算等）。
- **Reality constraints（现实约束）**：最常见的施工限制/物流限制（最小起送、短载费、搬运量、工作时间窗）。
- **Failure modes（失败模式）**：为什么会失败、失败通常不是“算错”，而是流程/材料选择/基层问题。
- **Measurement pitfalls（测量坑）**：哪里会导致数量级偏差（坡度、局部低点、非均匀厚度、包边/开孔）。
- **Materials differences（材料差异）**：同类材料的差异（密度、含水率、颗粒级配、规格覆盖差异）。
- **Checklist（清单）**：工具/配件/辅材清单（不堆品牌，不写泛泛“准备工具”）。

## 4) 本地推进流程（批量推进 + 统一检查）

建议每批处理 4 个计算器：

1. 选本批的 4 个（按审计 `thinTop` 顺序）。
2. 每个计算器：
   - 保留必备数组完整
   - 额外挑选 3–5 个模块（每篇组合不同）
   - 字数目标按 A/B/C 档区间走（不要刻意对齐）
3. 批量完成后统一检查：
   - `npm run content:check:calc`
   - `node scripts/local-duplicate-audit.mjs`
   - `npm run build`

## 5) 配置文件（可选，但建议）

如果需要“制度化”字数区间与差异化模块：

- 配置：`docs/calculator-content-variants.json`
- 审计读取：`node scripts/local-calculator-content-audit.mjs --minWords=500 --variants=docs/calculator-content-variants.json`

这个配置不是为了卡死内容，而是为了**避免全站出现“同字数、同结构”的统计特征**。
