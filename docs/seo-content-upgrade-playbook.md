# 网站增强（SEO + 低价值规避）通用方案（可复用）

这份文档总结的是我们在 HomeDecorCalc 上跑通的“长期、可持续、可批量推进”的改造方法，目标是：**提升内容实用性与独特性**、降低重复/低价值信号、并且每次上线都可被快速审计（本地 + 线上）。

---

## 0) 核心原则（避免跑偏）

- **以用户任务为中心**：页面要解决“我到底要买多少？怎么量？哪些坑会导致少买/多买？”而不是堆关键词。
- **计算器做“结果 + 决策支持”**：不仅给数字，还要解释假设、单位换算、常见浪费范围、购买单位（箱/袋/捆/张）与“为什么要向上取整”。
- **去重靠“分层”不是“删内容”**：重复的通用解释放到少数高质量的 `resources`/`guides`（pillar），计算器页只保留与该计算器强相关的内容，并通过内链引导。
- **宁可少做，但做一个成一个**：每个批次都必须通过 `audit:local` + `audit:live`，再继续下一批。

---

## 1) 选题与优先级（怎么选“最优”）

### 数据来源

- Google Search Console（GSC）：按查询/页面看 **impressions 高、CTR 低** 或 **排名 4–15** 的页面优先（最容易通过内容与标题/描述改善）。
- 站内结构：`/calculators/`（转化核心）优先于其它内容页。
- 站内审计脚本：薄内容、重复 title/description、canonical 异常优先处理。

### 选择策略（建议固定成 SOP）

1. 先选 **10 个页面**（一个小批次，避免反复提交）。
2. 每个页面只做“对准用户意图”的增强：
   - title/description 对准查询意图（不堆词）
   - FAQ 增强（公式、规则、边界条件）
   - 增加 1–3 条有用内链（到 pillar resource/guide）
3. 遇到“通用内容”重复出现时：**优先合并成 pillar**，原小页做 308/301 永久重定向。

---

## 2) 计算器页增强模板（每个计算器都按这套做）

把计算器页当成一个“可被引用的答案页面”，最少包含：

### A. SEO 基础（必须）

- `metaTitle`：包含用户常搜词（自然语言），但保持可读、不过长。
- `metaDescription`：一句话说明你能解决什么 + 包含关键购买单位/关键变量（例如 bundles/squares、sheetrock、cubic yards）。
- canonical/hreflang：保持正确（由站点统一逻辑保证，但审计要核对）。

### B. 页面上方内容（用户一眼能用）

- **3–6 条 Quick guide**：最关键的测量步骤与取值建议（例如“深度用平均值”“先确认包装覆盖率”）。
- **How to measure**：告诉用户如何测量输入项（每个输入至少 1 条）。
- **Assumptions to double-check**：列出 3–6 个最常导致结果偏差的变量（浪费、布局、间距、厚度、材料规格）。
- **Buying tips**：面向采购决策（向上取整、同批次、留修补量、特殊部位单列）。

### C. FAQ（强内容的关键区）

FAQ 不要写“废话”，建议每条都满足至少一个：

- 明确公式/换算（例如 `yd³ = ft³ / 27`、`sheets = area / sheetArea`）
- 给典型范围（浪费 5–10%/15% 的适用条件）
- 解释“为什么”（例如“为什么要加 waste”“为什么按 bundles 买”）
- 给“项目依赖”的分叉条件（例如 1/2" vs 5/8" 用在什么情况）

### D. 内链（少而精）

- 每个计算器页加 1–3 条“下一步”内链到：
  - `resources`：覆盖率表、换算表、材料清单
  - `guides`：完整流程、施工步骤、如何减少浪费
- 避免堆一堆同类链接；宁可少但相关。

---

## 3) Guides/Resources 的“pillar 化”策略（避免重复页）

### 什么时候会出现重复页？

当你以每个计算器为起点扩容时，最容易重复的是：

- 单位换算与常识（yd³、bundles/squares、sheet sizes）
- 通用浪费规则（对角铺、边框、切割）
- 通用材料清单（工具、准备、注意事项）

### 解决办法（推荐）

- 把这些“通用解释”集中到少数 **pillar resources/guides**。
- 其它微小页面不要保留成独立页面：合并到 pillar，然后用 **永久重定向（308/301）** 把旧 slug 指向 pillar。
- 确保：
  - sitemap 与列表页只保留 pillar
  - 内链全部更新到 pillar（避免重定向链）

（我们已经用 `docs/resources-consolidation-plan.md` 走完过多批次合并，可复用同样流程。）

---

## 4) 折叠长内容（体验 + 收录两不误）

目标：**默认展示一小段**（首屏更清爽），其余可展开阅读；同时避免“把内容藏起来导致不计权/不抓取”的风险。

建议做法：

- 首段（或首 120–200 词）始终可见。
- 其余内容使用“视觉折叠”（如 `max-height + overflow:hidden` 或 `line-clamp`），而不是把内容从 DOM 移除/异步加载。
- 展开按钮用可访问的 `aria-expanded`，并且无需额外请求即可显示完整内容。

---

## 5) 重定向是否影响 SEO？

永久重定向（`301`/`308`）通常是 Google 推荐的迁移方式：

- 会把旧 URL 的信号逐步转移到新 URL（需要时间）
- 迁移期可能有短期波动（正常）

上线前后建议：

- 避免重定向链（A→B→C）
- 更新 sitemap/内链到最终 URL
- 保证 canonical 指向最终 URL
- 用 `audit:live` 检查 `canonicalMismatchCount` 与非 200

---

## 6) 批量推进流程（固定成流水线）

每批建议 10 个页面，流程固定：

1. 选 10 个目标（GSC + 本地审计 + 转化优先级）。
2. 按“计算器页增强模板”修改内容（避免复制粘贴）。
3. 本地检查：
   - `npm run build`
   - `npm run audit:local`
4. 推送上线（同一批一起 push）。
5. 线上检查：
   - `npm run audit:live`
   - 额外抽查 3–5 个改动 URL（title/desc/canonical/hreflang/渲染/重定向）
6. 记录本批次改动点与对应 GSC 查询（方便复盘）。

审计结果文件位置：`calc-site/docs/audits/`。
